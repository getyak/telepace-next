# Epic: 坐实"Agent-first · 按用量 · 洞察层收费"的基础设施定位

> 类型：Epic / 里程碑集合　｜　建议并入 `TASKS.md` 的新 `T-6XX（商业模式）` 块
> 关联：`docs/seo-design.md`（T-5XX）、`telepace-vs-listenlabs-PRD.md`
> 更新日期：2026-07-22

## 背景

对标 Listen Labs（$500M 估值 / $20K·年 + $300–400·场 / 封闭 SaaS）正面打无胜算。telepace 唯一真差异化的方向是**开放、可编程、Agent-first 的研究基础设施**。定价必须成为这个定位的证据：**按用量/结果计费、PLG 自助、把钱收在"洞察 + 协作 + 复发"层，而非座位或 MCP 调用**。本 epic 定义从现状到该形态要完成的任务序列。

## 北极星 / 整体验收（Definition of Done）

- 计价主单位 = **一场"完成且达标"的访谈**（结果计费），座位/ MCP 调用**不**计费。
- 三档形态跑通：Free/Developer（生态 land）→ Pro（用量）→ Team（expand）。
- 付费墙卡在**洞察引擎 + 协作/集成**，执行层接近成本换量。
- 有一份经真实用户验证的付费意愿结论（而非假设）。
- `/pricing` 与 `compare/listen-labs` 上线，叙事对撞 Listen Labs。

## 排序原则（重要）

1. **Phase 0 验证是闸门**：`T-602` 未拿到正向结论前，不启动 Phase 2 的重型计费开发。
2. **先有能卖的东西**：`T-611` 洞察引擎是 PRD 头号缺口，没有它一切定价都是空管。
3. 依赖关系见每条 `依赖:`。

## Phase 0 — 验证与定义（低成本，去风险）

- [ ] **T-601 · 定义计价单位与"达标访谈"质量门**
  - 目标：产出一份可判定的"合格完成访谈"定义，作为计费与质量的共同底座。
  - 做：明确哪些访谈计费（完成度、时长下限、覆盖 key questions、非敷衍/非机器人），哪些不计；列出判定所需字段。
  - 验收：一份质量门规则文档 + 字段清单，工程可据此实现、任意一场访谈可被明确判为达标/不达标。

- [ ] **T-602 · ICP 与付费意愿验证（去风险闸门）**
  - 目标：用最小成本确认"AI-native 团队愿意为接进工作流的持续研究付费"，或及早证伪。
  - 做：锁定 1 个具体 ICP；与 ≥8 个目标客户做问题访谈 +/或 定价落地页测试；记录触发场景与可接受价格区间。
  - 验收：一份结论——willingness-to-pay 区间 + 复发触发场景，或"此楔子不成立"的证伪结论。**结论决定是否放行 Phase 2。**

- [ ] **T-603 · 定价决策记录（ADR）**
  - 依赖: T-601, T-602
  - 目标：把方向锁成可评审的决策，避免反复。
  - 做：写一份 ADR（用 `engineering:architecture` 体例），含：选定"用量/混合 + 洞察层收费 + 自带受众优先"、**被拒方案**（座位制 / MCP 调用计费）及理由、脉冲式用量风险与对冲。
  - 验收：`docs/` 下 ADR 一份，团队评审通过。

## Phase 1 — 价值捕获层（不建这层，什么都卖不出去）

- [ ] **T-611 · 洞察 / 报告引擎**（PRD §2.5 头号缺口）
  - 目标：把洞察卡片升级为可读叙事报告（章节 + 图表 + 证据可回溯），让"完成访谈"产出可付费的洞察。
  - 做：报告结构（叙事 + 内嵌图表 + 分群交叉）；数字/引用可点击回溯到受访者原话。
  - 验收：任一研究可生成结构化报告，引用可回溯；全绿。

- [ ] **T-612 · claim ↔ evidence 证据底座**
  - 依赖: T-611
  - 目标：建立"每个洞察/数字挂 N 条证据 id"的数据结构，支撑可追溯 + 置信度 + agent 引用（一次投入三处受益）。
  - 验收：数据模型落地，报告与洞察至少一处消费它。

- [ ] **T-613 · 访谈质量门落地**
  - 依赖: T-601
  - 目标：系统按 T-601 定义自动判定并标记每场访谈"达标/不达标"。
  - 验收：每场访谈有质量判定字段；不达标者可被计费系统排除。

## Phase 2 — 计量与套餐（Phase 0 放行后再启动）

- [ ] **T-621 · 用量计量（主单位 = 达标访谈）**
  - 依赖: T-613
  - 目标：按"完成且达标访谈"精确计量，语音分钟作为成本项单独记录。
  - 验收：每工作区可查询周期内达标访谈数 + 语音分钟，数据准确可对账。

- [ ] **T-622 · 套餐与配额执行（Free / Pro / Team）**
  - 依赖: T-621
  - 目标：三档计划 + 配额/超量逻辑落地，免费档封顶，付费墙卡在洞察/协作层。
  - 验收：各档限额可配置并强制执行；越限有清晰升级引导。

- [ ] **T-623 · Free/Developer 档 + MCP 获客动作**
  - 依赖: T-622
  - 目标：零摩擦免费档（N 场/月 + MCP + BYO LLM + 数据自持），作为生态 land 入口。
  - 验收：免费用户可自助接 MCP 跑访谈、数据归属清晰、达上限引导升级。

- [ ] **T-624 · 计费接入（credits / 用量，支持年度包）**
  - 依赖: T-622
  - 目标：打通支付，支持按达标访谈打包购买 + 低平台底费 + 企业年度承诺包（对冲脉冲式用量）。
  - 验收：可完成一次真实购买与超量计费；年度包提供可预测性。

  - 验收：自带受众端到端跑通；panel 仅占位不实现。

## Phase 4 — 扩张层（后置）

- [ ] **T-641 · 团队协作 + 深集成价值墙**
  - 目标：协作座位 + Notion/Linear/Slack + SSO + 自托管做成 Team 档锁定点。
  - 验收：这些能力仅 Team 档可用，形成 expand 抓手。

- [ ] **T-642 · 持续闭环（复发用量）**
  - 目标：把"信号触发 → 自动发起定向访谈 → 结果回灌"做成可用闭环，让用量复发。
  - 验收：至少一个 触发→访谈→回流 的闭环 demo 可跑通。

- [ ] **T-643 · （后置）Panel 加成 marketplace**
  - 目标：需求验证后再接第三方 panel，按参与者透传 + 加价。
  - 验收：写明启动的触发条件；**当前不做**。

- **M4 可讲清**：T-631~T-633，定价页 + 对比页上线。
